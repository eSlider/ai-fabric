## Possible Solutions
- Option A: Implement CI-driven semantic versioning on merge to `main` by calculating next version from commit metadata (or PR labels), creating a Git tag, and building/publishing artifact with that version.
- Option B: Keep versioning manual by requiring explicit version bump (e.g., `VERSION`/`package` file) in each feature PR, then build artifact from the committed version when merged.

## Recommended Approach
- Why this option is preferred
  - Choose Option A because it removes manual version conflicts across parallel branches, aligns version creation with merge-to-`main` (single source of truth), and directly supports production version switching via immutable tags/artifacts.
- Risks
  - Incorrect bump level if commit/label conventions are not enforced; tag creation race conditions on concurrent merges; rerun/idempotency handling in CI must be explicit.
- Dependencies
  - CI pipeline updates in `.gitea/workflows/`; version-resolution script (likely under `bin/`); repository permissions for tag push; agreed bump policy (Conventional Commits or PR labels); existing quality gates (`fmt`, `lint`, `test`, policy checks) remain green before release artifact step.

## Estimation
- Complexity: M
- Estimated effort: 1-2 days (policy + CI wiring + tests + dry run)
- Test scope: TDD for version resolver logic, CI integration test for merge-to-`main` flow, and validation that artifact name/tag/version are consistent and reproducible.

## Impact
- User impact
  - Predictable, traceable releases with clear semantic versions; easier rollback/forward and controlled prod version switching.
- Delivery impact
  - Slight PR discipline increase (message/label quality), but fewer manual release chores and fewer version conflicts.
- Operational impact
  - Requires maintaining tag/version policy and CI credentials; improves release auditability and artifact lifecycle management.
