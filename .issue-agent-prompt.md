- Labels update on issue creation

## Possible Solutions
- Option A: Set labels during issue creation by mapping classified type (`bug`/`feature`) to existing Gitea label IDs and passing them in the create-issue API call.
- Option B: Keep issue creation unchanged, then run a second post-create step to resolve label by name and update the issue labels via an additional API request.

## Recommended Approach
- Why this option is preferred  
  Choose Option A. It is simpler, atomic, and avoids temporary unlabeled issues; it also minimizes API calls and keeps logic in one path.
- Risks  
  Label name/ID drift across repositories, missing labels in target repo, and classifier output outside expected values (`bug`/`feature`) can cause unlabeled creations unless guarded.
- Dependencies  
  Gitea API support for labels on create, repository labels pre-provisioned (`bug`, `feature`), and existing issue-handler classification flow where issue type is decided.
