## Possible Solutions
- Option A: Extend `bin/telegram_bot.py` with a stateful `/projects` flow using Telegram inline keyboards and callback queries: fetch user-accessible Gitea projects, show selectable menu with current project marker, allow “Create new project”, persist selected project per chat/user, and make `/task` create issues in the selected project/repo context.
- Option B: Keep Telegram bot thin and add an internal “project context” service/script (used by bot) that handles list/select/create in Gitea plus persistence; `/projects` and `/task` call this service for all project logic.

## Recommended Approach
- Why this option is preferred: Choose Option A for fastest delivery and least moving parts in this PoC; it reuses existing polling, auth, and Gitea request patterns in one place and minimizes integration overhead.
- Risks: Callback-query handling is currently absent (bot only reads `message` updates), so adding interactive menus requires update parsing changes; project access filtering must align with Gitea permissions; context persistence must avoid cross-chat/user leakage.
- Dependencies: Gitea API endpoints for org/user repos and repo/project creation; Telegram Bot API inline keyboard + callback query support; lightweight persistence mechanism for selected project (file/db/env-backed runtime state), plus `/task` flow update to consume selected project.
